11-1 枚举其实是一个类

枚举可以用来表示不同的类型。将一个类继承 Enum，它就会变成一个枚举类。

  • 枚举的标签和数值是一对一的关系

如果 Vip 是一个普通的类，那么打印出的结果应该是1，2，3，4。

但是实际上打印出的就是变量名，这也正是枚举类的意义所在。我们看代码的时候，其实并不关心 YELLOW 的数值是多少，YELLOW 的数值是 1 或者 2 或者 3，对于写代码或者读代码来说是没有任何意义的，因为我们最直观的要的是类型，我们看到 YELLOW 这个词就知道它指代的是黄钻类型，看到 GREEN 我们就知道是绿钻，对于普通的 Enum 来说，我们知道变量背后所代表的意义就行了。

枚举的意义重在它的标签（YELLOW、GREEN、BLACK、RED就是标签），而不是在于它的数值（1、2、3、4是它的数值）。实际上我们可以将这些数值替换成任意的数据类型，只要满足不相同即可。比如换成：'a'、'b'、'c'、'd'。

11-2 枚举和普通类相比有什么优势

字典、普通类与枚举相比的缺点：

  • 字典、普通类是可变的；
    也就是说我们可以在代码里修改字典的值或者普通类变量的值，但是类型一但被确定下来之后是不应该被改变的。这是我们在定义枚举变量的时候使用全部字母大写的意义，大写表示常量，常量是不应该被改变的。
  • 字典、普通类没有防止相同值的功能；
    字典的 key 虽然不可以重复，但是字典的 value 是可以重复的，还有普通类的值也是可以重复的，但是实际上我们所要创建的类型是不允许重复的，比如我们用黄钻这个词来表示黄钻类型，又冒出个绿钻也可以表示黄钻类型，很显然会混淆视听。

显而易见，枚举类的优势：

  • 不可变
  • 可以防止重复

知识扩展：

枚举类型在 Python 里是如何实现的？

枚举类型在 Python 里是以一种单例模式实现的。也就是说对于枚举类型来说，你是不能进行实例化的。这也是枚举类象相对于普通类的区别。对它实例化是没有意义的，因为它是使用单例模式实现的。

23中设计模式：单例模式

对于想了解设计模式的同学，不建议去买设计模式的书去看，然后使劲的在那里看，这个是不建议的。设计模式的出现是为了解决软件工程里频繁变化的问题而产生的一种解决方案。大家还是需要把它带入到实践中去学习设计模式。换句话说，如果你问设计模式是不是必须的？那么我认为它不是必须的。

以我的经验来看，如果你在一个小公司到处使用所谓的设计模式，反而会对项目的工程造成负面的效果。

为什么呢？因为项目的成员或者你的同事可能并不懂你写的所谓的设计模式。设计模式的出现是为了防止你的代码出现频繁的变化，但是在一家小公司里，代码重新写、重新重构或者改的乱七八糟，这种情况实在是太正常了。这不是说是一个人能去解决的，这需要团队的所有成员对代码的质量有个共同的认识。如果说你尝试用设计模式的方式去解决团队代码混乱的问题，我觉得这不是一个好的办法，因为难度实在是太大了。

你看到很多人跟你吹设计模式多好多好，但我觉得实际情况无非是两种：

  1. 根本就没用到设计模式，只不过大家都说设计模式好，然后呢为了提升自己的格调，所以说他也说设计模式好；
  2. 确实是用到设计模式，而且设计模式也帮助到了他的团队解决很对问题。但是前提条件是他团队的每个人都很优秀，都对代码的质量和美观有一定的认识。在这样的前提条件下面设计模式确实是非常有用的。

我的建议是在编程的初期不要过多的关注设计模式，等你写了两三年、三四年代码之后，如果你确实是一个不满足于只实现业务，而是会经常去关注你自己写的代码质量高不高，如果你确定你是这样的程序员的话，那么设计模式就比较适合你，因为它确实可以帮你解决很多烦恼和问题。 如果你写了几年代码，你还是只关注业务功能的实现，仅仅是把业务功能写完就完事了，那么我觉得设计模式可能真的与你无缘了。

你要知道把一段业务逻辑平铺直叙的直接写下来，和用这种设计模式来构建代码，它们在初始阶段所花的成本绝对是不一样的，设计模式的优势只有在日后的很长一段时间之内才能体现出来。所以说你要衡量一下所花的时间成本到底值不值得。有时候出现了一个业务的变更，我们只需要改一个变量就能解决问题，如果说我们套用设计模式的话还需要给它补一个类，因为我们说过了扩展式开放的，修改是关闭的，如果严格按照这样一种设计模式来的话，有时候是得不偿失的。

最后一个建议：

如果你是一个做产品的，我说的这个产品有点类似与平台而不是说项目，就是说你做的项目是可以给别人提供服务的，对于这样一种产品呢，我建议大家使用设计模式，如果你只是写一个小项目呢，我并不建议大家过多的纠结于设计模式。

因为对于项目来说，它的生命周期并不长，特别是在现在比较快节奏的时代，我说的项目可能并不是大家理解上的项目，我说的项目是那种纯粹靠业务逻辑支撑起来的项目，没有太多技术含量，像这样的项目确实不建议大家使用设计模式。另外一种平台化的东西，相对于项目来说生命周期确实要长一些。

11-8 进阶内容开场白

在学习枚举知识之前我们已经可以利用所学习的知识来构建任何你想构建相关问题的解决方案了，因为在 Python2 的时候是没有枚举类型的，是在 Python3 才添加上去的。所以说枚举并不是一个必须、必不可少的解决问题的语法。这个和我们的类和函数是不太一样的，类和函数如果你不会的话，我觉得代码是没办法写下去的。其实也不能说的这么绝对，即使是你不懂类和对象的话呢，你只有函数和模块依然可以编写代码解决问题，只不过只用函数和模块，它不适合构建大型项目，所以说你加入类和对象之后就不一样了，可以把功能用面向对象的方式去构建，这样比较适合我们去构建一些中大型的项目。

那么问题来了。

既然我们掌握这些基础的知识之后就可以编写代码来构建我们的应用程序，那么我们为什么还需要一些高阶的知识来支撑我们的代码呢?

我们讨论这个问题必须要跳出语言的范畴，它并不是指 Python 一门语言是这样的。我们讨论的是编程的基础知识与高阶知识关系的问题。基础知识可以让我们写出代码，但是基础知识并不能让我们写出优质的、可维护的代码。

高阶知识主要用于编程的哪些方面？

  • 如果你只是一个业务流程或者说是业务逻辑的开发者，不考虑太多封装性，那么我觉得高阶知识对你来说可能用处并不大；
  • 如果你是包、类库的开发者，拟开发的包、类库需要提供给其他人来使用的话，那么高阶知识对你来说就是非常有用的。

11-9 一切皆对象

一切皆对象

函数式编程的概念

在其他语言里：函数只是一段可执行的代码，并不是对象，不能实例化。

在 Python 里：函数是对象，对象是可以实例化的。

函数

作为变量
  • 可以将函数赋值给一个变量
作为参数
  • 可以将函数作为另一个函数的参数
作为返回结果
  • 可以将函数作为另一个函数的返回结果

11-10 什么是闭包

闭包三要素

闭包必须满足三个条件：

  1. 必须是一个嵌套函数
  2. 嵌套函数必须引用一个外部的、非全局的、局部变量
  3. 闭包必须返回嵌套函数

满足这三个条件之后，Python会将嵌套函数、环境变量绑定在一起成型闭包，以后无论何时调用该闭包函数的时候，该环境变量都保持不变，因为该环境变量已经被绑定给嵌套函数了。

闭包 = 函数 + 环境变量（嵌套函数定义时的外部变量，且不能是全局变量），闭包示例如下：

闭包的意义

闭包保存的是一个环境，它把函数当时调用时候的现场保存起来了，并不仅仅只是返回函数。

这一点对于函数的调用来说是非常重要的。如果你只返回函数，而不返回函数调用时现场的相关环境变量的话，那么你的函数调用结果其实是非常容易受外部变量影响的。也就是说你很难保证你函数运行的正确性。

闭包并不是必须的。它只是说可以帮助你去精简或者说让你在代码的架构上面更加的合理，它是一种辅助的编程方式。

闭包只是一种思维方式的不同，是属于函数式编程的一种。

非闭包情况

  • 引用的变量在嵌套函数内部，这不是闭包，相当于就是返回一个函数，该函数能正常得到它想要的所有参数，并没有形成闭包；

  • 引用嵌套函数内部变量且存在同名全局变量，因为 a 就在嵌套函数内部，所以该嵌套函数能得到它所需的所有参数，并不需要去外部环境寻找 a，所以全局变量 a 的定义不影响结果；

  • 引用的是外部的全局变量，因为嵌套函数在内部找不到 a，会一层一层往外找，最后找到全局变量 a，因为没有形成闭包所以不会绑定外部的环境变量 a = 10，所以 a 的值会被后面的赋值表达式改为 5，调用的时候 a = 5

七月心语：

大家觉得函数式编程怎么样？我相信绝大多数同学都还是会感觉到困惑，会觉得这样一种编程体验非常烧脑子非常糟糕，这个并不是说函数式编程不好，其实我们这样的一种函数式编程的方式比起直接用全局变量的方式还是要好很多的。如果大家读过《代码大全2》这本书，其实这本书或者是很多的编程教科书里面都强调过经量少的去使用全局变量，代码里过多的使用全局变量是一个非常糟糕的行为，上例中你是在 go 函数内去改变全局变量，你怎么能保证其他的函数不回去保证全局变量呢？每一个函数都有可能改变全局变量，这就将导致你的函数没有自封闭性，它无法保证它自身的结果，它非常容易受另外函数的影响，如果它受其他函数的影响的话，那么这个函数的意义根本就是无从谈起的。

闭包的实现就好了很多，没有使用全局变量。

11-16 小谈函数式编程

七月心语：

我大概接触闭包比较多的有两个语言，一个是 Python，另外一个是 JavaScript。Python 对于闭包的定义强调的是环境变量。JavaScript 对于闭包的切入点，有个很重要的一点，它认为闭包可以实现在一个函数的外部间接的调用函数内部的变量（在外部函数的外部调用外部函数的变量，这个变量也就是我们所说的内部函数的环境变量）。

闭包保存环境变量的问题，它会导致什么问题呢？因为你的环境变量一直是常驻内存里的，它极容易造成内存泄漏，所以说大家在使用的时候一定要注意。因为你的环境变量为了保持它的状态，它肯定是要常驻内存的，所以说对于常驻内存的变量或者函数的定义，大家一定要小心使用。

内存泄漏的问题，在服务器里面其实相对还好，但是如果你是在 JavaScript 里操作浏览器的时候，由于环境变量常驻内存，非常消耗内存，极容易引起浏览器的卡顿和缓慢，会影响用户体验。

函数式编程没有好与不好，它有它的优点，也有它的缺点。闭包只是函数式编程的一种。我们后面讲 lambda 表达式的时候，大家会看到，函数式编程的思想也可以在我们现在主流思想里得到很好地应用，确实函数式编程有时候使用起来确实是很方便的。所以大家要区分开，闭包不等于函数式编程，闭包只是函数式编程的一种应用。
